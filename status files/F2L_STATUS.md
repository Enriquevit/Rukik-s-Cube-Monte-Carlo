# F2L Solver — Status & Technical Reference

## Overview

**F2L (First Two Layers)** is stage 2 of the CFOP method. After the cross is solved (4 white/U-layer edges in place), F2L pairs each of the 4 remaining U-layer corners with its matching middle-layer edge and inserts them together into the correct slot.

### Cube3.py Orientation

In this project the cross is on **U (white)**. The "working layer" for F2L is **D (yellow)**, not U as in standard CFOP.


Standard CFOP literature assumes cross-on-D with U as the working layer. Every algorithm is converted with the mapping `U→D, U'→D', U2→D2` (other faces unchanged).

| Slot | Corner Idx | Corner Faces | Corner Colors | Edge Idx | Edge Faces | Edge Colors |
|------|-----------|--------------|---------------|----------|------------|-------------|
| FR   | 4 (DFR)  | D, F, R      | Y, G, R       | 8 (FR)   | F, R       | G, R        |
| FL   | 5 (DLF)  | D, L, F      | Y, O, G       | 9 (FL)   | F, L       | G, O        |

### Last Layer (D face — not yet solved during F2L)

D-layer edges 4–7 (DR, DF, DL, DB) are the target of OLL/PLL and are left untouched during F2L.

## Architecture (F2L_solver.py)

### Components

1. **`classify_f2l_case(cube, slot)`** — Returns a case string `"c{pos}o{ori}_e{pos}o{ori}"` identifying where the target corner and edge currently sit and their orientation.

2. **`F2L_ALGORITHMS`** — Auto-generated at import time. For each of the 4 slots, builds a lookup table mapping `(corner_pos, corner_ori, edge_pos, edge_ori)` → move list. Generated by applying inverse algorithms from a bank of 35 base patterns × 4 D-premoves = up to 140 cases per slot.

3. **`solve_f2l_slot(cube, slot, solved_slots)`** — Two-phase:
   - **Fast path**: Static table lookup with D-premove variants. Instant.
   - **Fallback**: IDA* (iterative-deepening DFS) up to depth 10, first with slot-specific moves (3 faces), then all non-U moves.

4. **`solve_f2l(cube)`** — Solves all 4 slots sequentially: FR → FL → BL → BR. Each step preserves cross and previously solved slots.

5. **`F2LSolver` class** — Thin wrapper for visualizer compatibility.

### Move Sets

| Slot | Available Moves (slot-specific)     | Wider Fallback      |
|------|-------------------------------------|----------------------|
| FR   | R, R', F, F', D, D'                | All except U moves   |
| FL   | F, F', L, L', D, D'                | All except U moves   |
| BL   | L, L', B, B', D, D'                | All except U moves   |
| BR   | B, B', R, R', D, D'                | All except U moves   |

### Slot Rotation System

All 35 base algorithms are written for the FR slot. For other slots, a rotation mapping translates face names:

| Target Slot | Rotation (FR → target) |
|-------------|------------------------|
| FR          | identity               |
| FL          | R→F, F→L, L→B, B→R    |
| BL          | R→L, F→B, L→R, B→F    |
| BR          | R→B, F→R, L→F, B→L    |

### The D-Premove Technique

Before applying a table algorithm, the solver tries 4 D-layer adjustments:
- No premove, D, D', D2

This multiplies each algorithm's coverage by 4, since the target pieces may approach the slot from different D-layer positions.

## How F2L Works (Conceptual)

### Goal
After the cross, 4 corner-edge pairs must be inserted into the 4 slots between the cross edges. Each pair consists of:
- A **corner** (e.g., DFR = corner 4: Yellow-Green-Red)
- A middle-layer **edge** (e.g., FR = edge 8: Green-Red)

### Strategy

1. **Find the pair**: Locate the target corner and edge anywhere on the cube.
2. **Get them to the working layer (D)**: If they're trapped in another solved slot, extract them first.
3. **Pair them**: Maneuver corner and edge so they're connected (facing the right direction).
4. **Insert together**: A short algorithm (typically `R D R'` or `F' D' F` variant) places both pieces simultaneously.

### Case Categories

| Category | Description | Typical Length |
|----------|-------------|---------------|
| Already paired on D | Corner and edge are joined, just insert | 3 moves |
| Corner in D, edge in D | Need to pair then insert | 5–7 moves |
| Corner in slot, edge in D | Extract corner, pair, insert | 7–9 moves |
| Both in wrong slots | Extract both, pair, insert | 9–11 moves |
| Edge flipped | Edge orientation wrong, extra setup | 7–11 moves |

## Solver Status

### What Works
- ✅ Case classification (`classify_f2l_case`)
- ✅ Static lookup table generation (~58 cases per slot, 4 slots)
- ✅ IDA* fallback with iterative deepening (up to depth 10)
- ✅ Slot rotation system (FR algorithms adaptable to all 4 slots)
- ✅ Cross preservation verified during solve
- ✅ Previously-solved-slot preservation
- ✅ Single-seed test: FR slot solved in 7 moves (`R F' D F D' R' D'`), 0.48s

### Known Issues
- ⚠️ **cube3.py geometry bug**: `MOVE_CYCLES` / `FACE_CYCLES` have an inconsistency that causes some standard CFOP algorithms to disturb pieces they shouldn't. T-perm × 2 ≠ identity (known failing test). This may affect F2L correctness for certain scrambles.
- ⚠️ **Static table coverage**: Only ~58 of the possible F2L configurations are in the lookup table. Rare cases fall through to BFS, which can be slow at depth 10.
- ⚠️ **Not integrated in visualizer**: The V key uses `set_f2l_state()` (sets the cube directly to an F2L-solved state via OLL/PLL inverses) instead of running the solver.
